# VERIFIER-AUDIT-PR8.md

**Decision:** APPROVE WITH COMMENTS

The decomposition of `WorkbenchSession` into a facade plus six controllers is a faithful, behavior-preserving refactor. Every public method is preserved with semantically identical bodies in the six new controllers, all 235 tests across the workspace pass at the head SHA both in CI (Node 22 and Node 24) and locally, and no PROJECT.md rules were touched. There are two real but non-blocking concerns the maintainer should record: (1) the constructor on `packages/runtime/src/runtime/session.ts:46` duplicates the `SessionContext` shape inline rather than naming it directly, which is a DRY smell with bounded silent-drift risk; and (2) the elevation of `RunLifecycleController.assertRunActive` from private to public to support sibling-controller calls is undocumented as an inter-controller contract. Both are clean follow-up items, not regressions, so the PR is mergeable as-is and the items should be tracked rather than blocked on.

## Layer 2 — Architectural acceptance (diff-visible)

**Facade public surface.** I extracted every public member from the original `session.ts` (parent commit `1421e5d`) and the new facade and compared them pairwise. The original class has exactly 25 public members (1 getter `runId`, the `snapshot` method, and 23 mutator/query methods) plus 2 private members (`assertRunActive`, `transitionRun`). The Builder prompt's count of "37 prior public methods" is incorrect; the actual count is 25. The facade in `packages/runtime/src/runtime/session.ts:68-195` exposes all 25 of those public members with structurally identical parameter and return types. The facade does add explicit `: void` return annotations where the original had inferred-void returns (e.g., `requestGate`, `replaceRuleSet`); since the inferred and explicit types are identical, this is cosmetic. The facade also drops the `async` keyword on `exportRunBundle`, `writeArtifactAsync`, `materializeArtifact`, and `span`, but the public return type stays `Promise<…>` and the bodies forward the controller's promise directly (e.g., `packages/runtime/src/runtime/session.ts:88-90`), which is semantically equivalent because no synchronous throw can escape the facade method ahead of the controller's promise. The CLOSEOUT script in evidence #7 verifies that `dist/runtime/session.d.ts` (after stripping comments and private declarations) and the entire `dist/index.d.ts` are byte-identical to the pre-refactor outputs — that is the strongest possible evidence that the public surface is preserved, and it is consistent with my line-by-line comparison.

**Controllers are strictly internal.** `packages/runtime/src/index.ts` is unchanged at 18 lines and re-exports only `./runtime/session.js`, `./runtime/workbench.js`, `./runtime/hydrate.js`, `./runtime/readiness.js`, `./runtime/graph.js`, `./runtime/types.js`, and the two named exports from `./runtime/state.js`. Neither `sessionContext.ts` nor any of the six `*Controller.ts` files appear in the export surface. The Builder honored the prompt's constraint that the controllers and `SessionContext` remain internal.

**Constructor type safety.** This is the maintainer's first pre-flagged concern and it is real but bounded. The constructor at `packages/runtime/src/runtime/session.ts:45-66` declares `ctx` with an inline anonymous type that is a structural copy of `SessionContext` at `packages/runtime/src/runtime/sessionContext.ts:5-18`, then assigns `const sessionContext: SessionContext = ctx;` on `packages/runtime/src/runtime/session.ts:59`. The drift risk is narrower than it might appear: TypeScript's structural assignability check on line 59 means that if a future PR adds a *required* field to `SessionContext`, the inline `ctx` shape will fail to be assignable and the constructor will fail to type-check, surfacing the drift loudly. The risk is real only for *optional* fields added to `SessionContext` — those would silently pass through `undefined` to all six controllers without the inline shape declaring them, and downstream callers of `WorkbenchSession` would have no way to discover the new optional configuration through the facade's constructor signature. There is also a DRY concern: two declarations of the same shape in adjacent files invite future skew. Set against this is the architectural reason the Builder kept the inline shape: changing the constructor to `constructor(ctx: SessionContext)` would either expose `SessionContext` in the public type surface (because TypeScript emits the resolved type into `dist/runtime/session.d.ts`) or require renaming `SessionContext` to a name the runtime consumers can already see — both are non-trivial decisions about what becomes public API. My judgment is that this does not rise to a blocking issue: the slice's primary goal (decompose without behavior change and without changing the declaration file) is correctly achieved, and the cleanest fix is a focused follow-up that explicitly decides whether `SessionContext` belongs in the public surface. I therefore record this as comment #1 below rather than as a REQUEST CHANGES blocker.

**Inter-controller dependencies.** The constructor at `packages/runtime/src/runtime/session.ts:60-65` wires the dependency graph: `lifecycle` is constructed first with no siblings, `gates` depends on `lifecycle`, `steps` depends on `lifecycle` and `gates`, and `artifacts`, `trace`, and `rules` each depend only on `lifecycle`. The graph is acyclic, only `lifecycle` is depended on by siblings, and `lifecycle` itself depends on nothing. This shape is sensible. The cost is that `RunLifecycleController.assertRunActive` is `public` at `packages/runtime/src/runtime/runLifecycleController.ts:17` so siblings can call it (e.g., `packages/runtime/src/runtime/gateController.ts:20`, `packages/runtime/src/runtime/stepController.ts:28`, `packages/runtime/src/runtime/artifactController.ts:30`, `packages/runtime/src/runtime/traceController.ts:30`, `packages/runtime/src/runtime/ruleController.ts:17`) — but it is not actually intended to be called from outside the runtime package. The risk is bounded by the fact that the lifecycle controller is `private readonly` on the facade (`packages/runtime/src/runtime/session.ts:38`), so external callers cannot reach it through the public surface. The remaining risk is internal: a future contributor adding a seventh controller could call `lifecycle.assertRunActive` correctly, but a future contributor refactoring the lifecycle controller might assume `assertRunActive` is unused outside this file because there is no JSDoc comment marking it as the inter-controller gate. This is a documentation gap. I record it as comment #2 below.

## Comments (recommended follow-ups)

1. **Resolve the inline-`ctx` / `SessionContext` duplication.** At `packages/runtime/src/runtime/session.ts:45-66` the constructor declares an anonymous shape that duplicates `SessionContext` from `packages/runtime/src/runtime/sessionContext.ts:5-18`, and line 59 asserts assignability between them. The duplication is bounded but unnecessary, and it leaves silent-drift risk for *optional* fields added to `SessionContext` in the future (required-field drift would be caught at compile time by line 59). Recommended action: in a small follow-up slice, decide whether `SessionContext` should be exported from `packages/runtime/src/index.ts` and the constructor changed to `constructor(ctx: SessionContext)` (which would surface the type in `dist/runtime/session.d.ts`), or whether `SessionContext` should remain internal and the constructor should be marked `@internal` because production callers go through `WorkbenchHost`. Either resolution is acceptable; the current state is not.

2. **Document the inter-controller contract for `assertRunActive`.** `RunLifecycleController.assertRunActive` at `packages/runtime/src/runtime/runLifecycleController.ts:17-25` is `public` so that `GateController`, `StepController`, `ArtifactController`, `TraceController`, and `RuleController` can call it as the standard run-active gate. There is no JSDoc comment marking this as the intended contract, which means a future contributor (a) refactoring `RunLifecycleController` may not realize the method has cross-file callers inside the runtime, and (b) adding a new controller may not realize calling `lifecycle.assertRunActive` is the established pattern. Recommended action: add a short JSDoc above the method along the lines of `/** Public so sibling controllers can gate writes on the run being active. Not part of the public WorkbenchSession surface — external callers reach this only through the facade methods. */`. Optionally encode this convention in PROJECT.md §Conventions as a forward-looking rule for future controller splits (see Methodology learnings below).

## Methodology learnings

The first methodology slice surfaces three things worth folding back into the loop.

The Builder prompt's count of "37 prior public methods" was wrong — the actual count is 25 public members plus 2 private. This did not cause a defect (the Builder preserved every member that actually existed), but a Verifier following the prompt literally would have hunted for 12 phantom methods. Future Builder prompts should derive the public-method enumeration mechanically (e.g., by running `grep` against the parent commit and listing each name) rather than asserting a count, or should pin the list of names with no count at all. The fact that the maintainer's prompt to me listed 25 names (matching reality) but said "all 37" indicates the count was a recall error rather than a methodology gap, but the Builder/Verifier loop benefits from removing both.

The Builder's "Findings deferred to follow-up slice" section in `CLOSEOUT.md` is currently scoped to "things I noticed but explicitly chose not to do." That framing missed both architectural choices the Builder made on its own initiative: the elevation of `assertRunActive` to public, and the inline-`ctx` constructor shape. Both were defensible decisions, but neither was flagged for the maintainer to ratify or reverse. Future Builder prompts should require a separate "Architectural choices" section in `CLOSEOUT.md` enumerating any decisions the Builder made that were not explicitly specified in the Builder prompt — even (especially) when the Builder believes the decision is correct. This makes the Verifier's audit cheaper because the choices are surfaced rather than hunted, and it makes the methodology loop tighter because the maintainer has a single section to scan for "did the Builder invent something?".

The pattern of "elevate a private method on one controller to public so siblings can call it" is likely to recur in the upcoming `WorkbenchShell.tsx` split and in any further runtime decomposition. PROJECT.md §Conventions does not currently address this. A short rule along the lines of "When a refactor splits a class into multiple controllers and one controller's private method becomes a cross-controller dependency, the elevated method must carry a JSDoc comment naming the controllers that may call it and explicitly stating it is not part of the public surface" would close the gap, eliminate the documentation gap I flagged in comment #2, and pre-empt the same finding on the `WorkbenchShell` split. The rule would be enforceable by the Verifier with a simple text search.
